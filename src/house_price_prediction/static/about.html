§ 05

Bias–Variance Tradeoff

Model performance depends on striking the right balance between two competing sources of error.

High Bias (underfitting)
model too simple
poor training performance
poor test performance
High Variance (overfitting)
memorizes training data
excellent train performance
poor generalization
Goal: A good model balances bias and variance to generalize well to unseen data.
§ 06

Regularization

Regularization prevents overfitting by penalizing large model coefficients. This project uses Ridge Regression (L2) as its main model. 

# Ridge (L2) loss function
Loss = Error + α Σ(w²)
shrinks coefficients reduces variance keeps all features
§ 07

Cross-Validation

Cross-validation evaluates how stable the model is across different data splits. GridSearchCV was used to automatically tune the best regularization strength (alpha). 

split dataset into k folds
train on k−1 folds
validate on remaining fold
repeat across folds
average results

§ 09

Log Transformation

House prices are often right-skewed. Applying a log transform to the target variable stabilizes variance and improves regression performance.

During training
y_log = log(y)
At prediction time
y = exp(prediction)
§ 10

Residual Analysis

Residuals (Actual − Predicted) reveal whether the model captures all signal in the data. An ideal residual plot shows random scatter with no visible structure. 

Ideal:   random scatter · constant variance · no structure

Warns:   model bias · missing features · heteroscedasticity

Training vs Inference Separation

In production systems, training and prediction must be fully decoupled. This separation ensures faster predictions, simpler deployment, and stable behavior.

Training pipeline

  ↓

  ↓

Serving system

  ↓

  ↓
return response

§ 14–15

Configuration & Dependency Injection

Hardcoded values are replaced with environment variables, enabling clean separation between development, testing, and production configurations. 


# Environment variables
MODEL_PATH   = "models/latest.joblib"
APP_ENV:      = "production"

# Dependency injection
class HousePriceModel:
def __init__(self, model):
        self.model = model

The prediction service receives the model as an external dependency rather than loading it internally — making components easier to test and swap.

§ 19–20

Docker Containerization

The inference service is fully containerized. The Dockerfile layers a Python base image, installs dependencies, copies source code and model artifacts, then runs FastAPI. Docker Compose simplifies local development to a single command. 

# Dockerfile steps
FROM   python:3.11-slim
WORKDIR   /app
COPY    pyproject.toml /app
COPY    src /app/src
RUN    pip install -e .
COPY   models /app/models
EXPOSE   8000
CMD    ["uvicorn", "house_price_prediction.api:app", "--host", "0.0.0.0", "--port", "8000"]

# Start everything
$ docker compose up

Frontend Interaction Flow

When a user clicks Estimate Price, a ten-step chain connects the browser to the trained model and returns a prediction. 

1  User enters house features
2  User clicks Estimate Price
3  JavaScript collects input values
4  JavaScript builds JSON payload
5  Browser sends POST /predict
6  FastAPI validates request
7  Prediction wrapper runs model
8  Model returns predicted price
9  API sends response
10 Frontend displays prediction
